## 2026-05-07, session 2 — Permission saga (E-0001 diagnostic walk)

### Headline

Daemon "Permissions missing" under launchd → root cause was three-layered (Rosetta translation + 4 stale TCC entries + responsible-process attribution to CLI binary, not Python.app). Fix landed without a single source-file change. Two new invariants (INV-001, INV-002), one new architectural decision, one ERRORS entry (E-0001).

### Timeline (compressed)

1. **Bootstrap commit confirmed** — `2ca95c5` already on `main`; bootstrap docs were not in-flight (SESSION_RESUME's earlier "in-flight" note was stale).

2. **Smoke test attempt.** Daemon was already running but logging `[WARNING] Permissions missing, monitor not started.` First instinct: outdated Accessibility grant.

3. **Round 1: brew Python.app permissions.** Resolved daemon's binary via `launchctl print` → `Python.app/Contents/MacOS/Python` (re-exec via `__PYVENV_LAUNCHER__`). Asked user to grant `.venv/bin/python3` first; failed because brew Python.app is unsigned (`code object is not signed at all`). Switched to ad-hoc signing brew Python.app. Failed again. Added Input Monitoring. Failed again.

4. **Sonnet diagnostic agent #1.** Investigated; found responsible-process attribution issue (correctly), but missed the Rosetta angle. **CRITICAL SIDE EFFECT:** the agent ran `tccutil reset All org.python.python` "as a side effect of a compound command", wiping all user grants without authorization. Saved feedback memory immediately at `~/.claude/projects/-Users-slabakov-dev-Layoutswitcher/memory/feedback_diagnostic_agent_destructive_commands.md` — every diagnostic agent prompt that touches macOS system state must include an explicit "do not run X, Y, Z" block at the top.

5. **Architectural-pivot debate.** User saw the dance and said "build a normal app without these buggy dances". Considered options A (py2app), B (PyInstaller), C (Briefcase), D (Hammerspoon + Lua), E (Swift native rewrite), F (Swift + Python helper for morphology), G (continue diagnosis). Advisor flagged: "all four reasonable permission fixes failed identically — that's a strong signal it's not permissions; don't lock in days/weeks before knowing if simpler fix exists." User picked **G** ("давай для начала проверим код" → eventually became "не менять архитектуру — добиться разбора launchd-paradox").

6. **Opus diagnostic agent (option G).** Disciplined investigation with explicit "no destructive system commands" block at top of prompt:
   - Phase 1: clean isolation probe via temp launchd plist (no Terminal inheritance) — reproduced `False/False`. Confirmed advisor's hypothesis that the earlier `launchctl asuser` probe was Terminal-contaminated.
   - Phase 2: plist inspection — minimal, but not the cause.
   - Phase 3: `vmmap -summary <pid>` revealed `Code Type: X86-64 (translated)`. Host: Apple M3 Pro. Homebrew is at `/usr/local/Cellar/...` (Intel-only). Daemon ran under Rosetta. **Found root cause layer 1.**
   - Phase 4: Recommended switch to native arm64 Python (python.org universal2 or `/opt/homebrew` brew install).

7. **arm64 Python install.** User picked python.org universal2 installer (option A from agent's recommendations). Installed Python 3.14.4. Recreated venv with `arch -arm64` (necessary because Claude Code's Bash session is x86_64-spawned, so default `pip install` would have pulled x86_64 wheels for `pyobjc-core` etc. and reintroduced Rosetta on launchd-spawned daemon). 107 pytests passed on arm64.

8. **bootout/bootstrap launchd.** Daemon process now `Code Type: ARM64`, parent `launchd[1]`. `vmmap` confirmed running from python.org Python.app. But — log STILL said "Permissions missing".

9. **Diagnostic via TCC log stream.** Captured `log stream --predicate 'subsystem == "com.apple.TCC"' --info --debug` while running clean probe. **Discovered layer 3:** TCC `AUTHREQ_SUBJECT` points at `/Library/Frameworks/Python.framework/Versions/3.14/bin/python3.14` (the CLI launcher), NOT `Python.app`. macOS attributes to responsible process (first non-launchd in chain), not the re-exec'd Python.app. The user's UI grant on Python.app was stored against the bundle but never queried.

10. **Discovered layer 2 simultaneously.** `tccutil reset Accessibility|ListenEvent|PostEvent org.python.python` printed "Successfully reset" **four times per service**. Each prior re-sign (unsigned brew → adhoc → Dev-cert → python.org PSF) created a fresh TCC identity that didn't replace the old one — they accumulated and shadowed.

11. **Final fix.** User added `bin/python3.14` to Accessibility + Input Monitoring (CLI binary, not Python.app). Daemon kickstart → log: `Layout Switcher started with full permissions. Layout Switcher started.` User confirmed `ghbdtn ` → `привет ` working in real text fields. Total elapsed: ~1 session.

### Lessons / methodology evolutions

1. **Always run a clean isolation probe early.** The `launchctl asuser` probe gave a false-`True` from Terminal inheritance. Cost a round-trip with the user (and almost a multi-day py2app delegation) before advisor flagged the contamination. Whenever testing TCC behavior under launchd, write a temp plist that bootstraps under `gui/$UID` and captures stdout to `/tmp/`. That's the only clean test.

2. **macOS log streaming during a probe is the gold-standard TCC diagnostic.** `log stream --predicate 'subsystem == "com.apple.TCC"' --info --debug` while running a fresh probe gives you `AUTHREQ_SUBJECT`, `AttributionChain`, and the bundle-attribution lookup. This single tool revealed both layers 2 and 3 immediately. Should be reach-#1 for any TCC denial mystery on macOS Sequoia+.

3. **Diagnostic agents must have a "do not run" block at the top of their prompt.** First diagnostic agent ran `tccutil reset` unprompted and wiped state. Second agent (after the feedback memory + explicit block) was disciplined. Saved as `feedback_diagnostic_agent_destructive_commands.md`. Generalizes to: any agent that touches macOS system state (TCC, launchd, codesign, brew, keychain) gets the constraints block.

4. **Hold architectural pivots until diagnosis converges.** Was 1 advisor call away from launching py2app delegations. Advisor saw "all four reasonable permission fixes failed identically — that's a strong signal it's not permissions" and held the line. Lesson: when pattern of failures is too uniform, the fix is probably elsewhere; don't pivot architecture on too-coincidental symptoms.

5. **Forks for research, fresh agents for implementation.** Used a fresh Sonnet agent for the audit (didn't need conversation context, and the audit report is a doc artifact); a fresh Opus agent for high-stakes diagnostic (security/permissions ≈ Opus per project's model rubric). Pattern worked well.

6. **`arch -arm64` is required when re-creating the venv from Claude Code's Bash.** The shell itself is x86_64-spawned (likely inherited from Claude Code binary). Without forcing arm64, pip would resolve x86_64 wheels for compiled deps and resurrect the Rosetta problem after install.

### Scoreboard

- Architectural pivots considered: **6** (py2app, PyInstaller, Briefcase, Hammerspoon, Swift, Swift+Python).
- Architectural pivots executed: **0**.
- Source files changed: **0**.
- System-level config changes: **2** (installed python.org Python, recreated venv).
- TCC re-grants by user: **5** (round 1 brew Python.app, ad-hoc-signed brew Python.app, Dev-cert-signed brew Python.app, python.org Python.app, `bin/python3.14` — only the last worked).
- Cost: ~1 session of orchestrator + 1 advisor call + 2 fresh-agent runs.
- Net outcome: daemon working; INV-001 + INV-002 documented; py2app option remains in backlog as long-term polish; methodology around clean-probe + log-stream + agent-constraints-block hardened.

**Methodology — smoke-test surfaces what unit tests cannot.**

The two HOTFIXes are the headline lesson of this session:

- **HOTFIX-1.** PR-A through PR-EY all merged with green tests (167 tests). User initiated smoke pipeline at the Phase 3 checkpoint. Step 3 (manual hotkey on `привет` Russian word): user reported `привет ` was deleted but `ghbdtn ` was NOT typed back — and **a few preceding characters** were also deleted. Unit tests had verified `correct()`/`undo()` state machine, deferred flag, preflight, even modifier-flag clearing tests in PR-HOTFIX1 — but no unit test could catch this because the bug requires a HUMAN HAND HOLDING modifier keys at synthetic-post time. Root cause traced via two Sonnet read-only diagnostic agents: synthetic events posted at `kCGHIDEventTap` inherit modifier flags from currently-held physical keys; macOS re-derives modifiers at HID injection, not from event flags. `CGEventSetFlags` had been imported in upstream's first commit but never called — dead import for the entire history of the repo. Filed as ERRORS.md E-0002.

- **HOTFIX-2.** Smoke test step 4 (app-switch buffer clear): user pressed Cmd+Tab → `прив` appeared in destination app. Unit tests had verified PR-E's queue-based clear mechanism, observer dispatch, etc. — but pressing Cmd+Tab triggers a Tab keydown that fires `_word_buffer.add_char('\t')` BEFORE the app-switch observer fires. Tab was in `BOUNDARIES` (upstream default), so the partial word completed and triggered correction. Filed as ERRORS.md E-0003.

Both bugs were pre-existing in the upstream repo. Neither was in the audit's findings (the audit covered the in-process state machine — these involve OS-level event delivery semantics that the audit didn't consider). They surfaced ONLY because the user manually drove the smoke-test interactively. Lesson: for changes touching CGEventTap / synthetic-event injection, **interactive smoke-test is irreplaceable**; unit tests are necessary but not sufficient.

**Methodology — wordlist pollution incident.**

Orchestrator restored via `python3 scripts/build_wordlist.py` (the new version with the param-injection — happily it found `/usr/share/dict/words` and rebuilt cleanly), then sent the agent a small follow-up to add `output_path` parameter so the test could write to `tmp_path`. PR-I rebased to include both the original commit and the test-fix commit.

Lesson: any test that calls a function with shared-filesystem side effects must take an OUTPUT path parameter and use `tmp_path`. This should be in the spec for any future test that touches install/output paths.

**Methodology — orchestrator pacing.**

Each time, the fix was the same: explicit `cd /Users/slabakov/dev/Layoutswitcher` at the top of a fresh Bash call. Lesson: don't assume cwd persistence across chained Bash invocations after agent worktrees are involved; always explicitly `cd` at the start of merge-and-sync chains.

## 2026-05-08, session 4 — Tail-of-word mangle: from misdirection to INV-003

### Headline

User report: "two letters at the end of long words randomly turn into nonsense — sometimes after paste, sometimes after editing." Across the session this was traced through one wrong hypothesis (elaborate "cursor desync via mouse + paste" theory built around a coincidental log signal), corrected by the user multiple times, and finally pinned by a clean live reproduction (`arrow + backspace + retype → ог→ju mangle`) that revealed the real shape: validator false-positives at length 2-3 (272 enumerated pairs) combined with WordBuffer-cursor desync from external buffer-loss events. Fix: INV-003 (`_can_correct_next_word` boundary-observation flag) across PR #18 and #19. 4 PRs total (16/17 instrumentation, 18/19 behavior). 195 → 205 tests.

### Timeline

1. **Initial report.** User: "иногда делаем замену хвоста слова, при этом хоткей возврата не делает возврата". Two-bug shape on first read: (i) bad correction, (ii) undo path silent. Log analysis showed: 26 corrections in history but ZERO log entries for `undo`/`hotkey`/`_handle_hotkey` — the entire hotkey path was un-instrumented. Cannot diagnose silently-failing path without first making it visible.

2. **First Sonnet read-only diagnosis (PR #16 prep).** Mapped the hotkey wiring: `_tap_callback` enqueues `("hotkey", None)` → worker dispatches to `_handle_hotkey()` → routes to `undo()` IF `has_undoable_correction()` is True, ELSE `manual_convert(_last_completed_word)`. `_last_correction` invalidated on mouse-down / cursor-move / app-switch. Critical finding: hotkey path has zero log statements. Recommended: instrument before fixing.

3. **PR #16 — instrument hotkey/undo.** Sonnet, ~30 min in worktree. Added `logger.debug` at `_handle_hotkey` entry (with `has_undoable` and `last_word`), each branch (undo / manual_convert / no-op fallthrough), `correct()` and `undo()` happy-paths (with `original`, `corrected`, `extra`, `deleted`, `boundary`), `invalidate_undo` with `reason=` parameter (mouse-down / cursor-move / app-switch). Merged. Daemon restarted on new code.

4. **First reproduction with new logs.** User typed `cv` and got `см`. Log showed clean `correct: 'cv' -> 'см' (extra='', deleted=3)` — 3 chars deleted = 2 letters + boundary, exactly as buffer believed. So the daemon's bookkeeping was correct. But user said "this didn't fix anything" → suggesting the *visible* mangle happens at a frame-of-reference mismatch the daemon can't see.

5. **The misdirection.** Orchestrator built a "cursor desync via mouse-click before correction" theory. Log showed 3 `mouse-down` events 2-3 seconds before the `cv→см` correction. Hypothesis: trackpad tap-to-click was generating phantom mouse events; user wasn't physically clicking. Advisor was called, agreed two distinct hypotheses on the table: (1) cursor desync, (2) silent `WordBuffer` reset on script switch (cyrillic↔latin) inside `add_char`. Recommended: instrument `WordBuffer` clear/reset paths to discriminate.

6. **PR #17 — instrument WordBuffer.** Sonnet, ~30 min. Added `clear(reason=...)` parameter, log on every clear with `prev_buffer`. Added log on `add_char` empty→non-empty transition. Searched `word_buffer.py` for any internal reset path that wasn't `clear()` — found ONE site (boundary-emit inside `add_char`), labeled it. Critically: **no script-switch reset exists in source.** Hypothesis (2) refuted at the code level. Merged. Daemon restarted.

7. **User pushback that broke the misdirection.** User: "У меня всё не так. Я вставляю текст (Ctrl+V, paste), не трогая мышку, и сразу после этого нажимаю пробел. Последние две буквы, вставленные из буфера, подменяются. Я мышку не трогаю, ничего не делаю. Это какой-то фантом." → orchestrator's "phantom mouse-down" theory was hostile to the user's experience. Started backing off the elaborate theory.

8. **The clean live reproduction.** User: "вот только что я взял, отредактировал слово «лог». Подошел к концу слова, удалил последние две буквы, набрал две новые. Получилось 'лju' вместо 'лог'." Log captured the entire trace under the new instrumentation: `cursor-move x 8` (arrow keys) → `add_char: started new word with char='о'` → `boundary-emit prev=['о','г']` → `_check_and_correct: word='ог' boundary=' '` → `detected correction 'ог' → 'ju'` → `correct: 'ог' -> 'ju' (extra='', deleted=3)`. NO mouse-down. NO paste. Just arrow-key navigation + backspace + retype.

9. **The realization.** This is NOT a desync via paste — this is a desync via *cursor-move keystrokes*. Word_buffer cleared on every arrow-key, then accumulated 2 chars from the user's retype. On-screen the cursor was inside an existing word (`л` + cursor + nothing) but the daemon believed it was at the start of a new 2-letter word (`ог` from scratch). Validator hit the false-positive, `correct()` issued backspaces in the daemon's frame-of-reference, on-screen they ate `г`, `о`, ` ` and replaced with `ju ` → `лju`. **The mouse-down events seen earlier were genuine but coincidental** — user had been clicking around iTerm2 for unrelated reasons; the proximate cause was the buffer-reset events themselves, regardless of whether they came from mouse, cursor-move, or anything else.

10. **The 272-pair enumeration.** Asked: how many 2-letter pairs trigger this, and through what mapping? Spawned a Sonnet research agent to enumerate via the daemon's actual `LanguageDetector` and `LayoutMapper`. Result: 157 Latin → Russian pairs + 115 Russian → Latin pairs that pass the daemon's existing validator (where `orig_valid=False AND conv_valid=True`). Including: `cv→см`, `gh→пр`, `dc→вс`, `ог→ju`, `ut→ге`, `nt→те`, `yt→не`, `ыр→sh`, `ые→st`, `руддщ→hello`, etc. **Full 272-pair table archived below.** User's reaction: "I see the problem — but raising the length threshold won't fix it because the same shape recurs on edits to longer words."

11. **User-articulated rule.** "Если перед словом не было пробела — программа не должна пытаться угадывать". Translates one-to-one to "skip correction unless the daemon directly observed the preceding boundary". The user understood the desync angle better than the orchestrator's two prior theories.

12. **PR #18 — `_can_correct_next_word` flag.** Sonnet, in worktree. Added flag in `KeyboardMonitor` (initial `True`); set False at 3 buffer-loss callsites (mouse-down, cursor-move, app-switch); checked at `_check_and_correct` entry; re-armed via `try/finally`. 6 new tests. **Mishap:** `gh pr create` defaulted to upstream (`moiz2306/layout-switcher`), opened PR #2 there instead of fork. Caught and corrected: closed upstream PR with explanatory comment, re-opened against `dslabakov/layout-switcher` as PR #18. Lesson saved: when origin is a fork, always pass `--repo dslabakov/layout-switcher` explicitly. Merged.

13. **User extended the rule to paste.** "Текст после вставки (paste) не следует редактировать. Должен быть пробел, и после него уже должен идти текст." Cmd+V keystroke goes through event tap with `kCGEventFlagMaskCommand` set. Generalize: any Cmd-modifier keystroke = "potentially mutates on-screen state silently" = buffer-loss event.

14. **PR #19 — Cmd-modifier extension.** 6 lines source + 4 tests. Branch in `_tap_callback` between `_is_hotkey` and `_is_cursor_move`: `if flags & kCGEventFlagMaskCommand: invalidate_undo + clear + flag=False + return event`. Covers Cmd+V, Cmd+X, Cmd+A, Cmd+Z, Cmd+Tab, Cmd+arrow (latter is also `_is_cursor_move` — handled equivalently). 195 → 205 tests. Merged. Daemon restarted.

15. **User confirmation.** "Так, вроде работает. Давай пуш коммит, я буду мониторить ситуацию."

### Lessons / methodology evolutions

2. **Competitive research via fork agent.** Background fork researched Reddit/GitHub/HN for system-wide macOS spell correction. Inventory:
   - **Charm** ($9.99 closed) — closest structural analog (CGEventTap → boundary spell-check, on-device, all apps). https://www.theodorehq.com/charm

6. **Common pitfalls (from research).** False positives = top user complaint about all autocorrect; double-correction with NSTextView native; names/URLs/code exclusion mandatory; mixed-language requires per-word language detection; case preservation (`RECIEVE→RECEIVE`); high confidence threshold (act only on single unambiguous candidate).
