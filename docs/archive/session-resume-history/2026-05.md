### Lessons / methodology evolutions

1. **Always run a clean isolation probe early.** The `launchctl asuser` probe gave a false-`True` from Terminal inheritance. Cost a round-trip with the user (and almost a multi-day py2app delegation) before advisor flagged the contamination. Whenever testing TCC behavior under launchd, write a temp plist that bootstraps under `gui/$UID` and captures stdout to `/tmp/`. That's the only clean test.

2. **macOS log streaming during a probe is the gold-standard TCC diagnostic.** `log stream --predicate 'subsystem == "com.apple.TCC"' --info --debug` while running a fresh probe gives you `AUTHREQ_SUBJECT`, `AttributionChain`, and the bundle-attribution lookup. This single tool revealed both layers 2 and 3 immediately. Should be reach-#1 for any TCC denial mystery on macOS Sequoia+.

3. **Diagnostic agents must have a "do not run" block at the top of their prompt.** First diagnostic agent ran `tccutil reset` unprompted and wiped state. Second agent (after the feedback memory + explicit block) was disciplined. Saved as `feedback_diagnostic_agent_destructive_commands.md`. Generalizes to: any agent that touches macOS system state (TCC, launchd, codesign, brew, keychain) gets the constraints block.

4. **Hold architectural pivots until diagnosis converges.** Was 1 advisor call away from launching py2app delegations. Advisor saw "all four reasonable permission fixes failed identically вАФ that's a strong signal it's not permissions" and held the line. Lesson: when pattern of failures is too uniform, the fix is probably elsewhere; don't pivot architecture on too-coincidental symptoms.

5. **Forks for research, fresh agents for implementation.** Used a fresh Sonnet agent for the audit (didn't need conversation context, and the audit report is a doc artifact); a fresh Opus agent for high-stakes diagnostic (security/permissions вЙИ Opus per project's model rubric). Pattern worked well.

6. **`arch -arm64` is required when re-creating the venv from Claude Code's Bash.** The shell itself is x86_64-spawned (likely inherited from Claude Code binary). Without forcing arm64, pip would resolve x86_64 wheels for compiled deps and resurrect the Rosetta problem after install.

### Lessons / methodology evolutions

1. **When user says "—Г –Љ–µ–љ—П –≤—Б—С –љ–µ —В–∞–Ї" вАФ the hypothesis is hostile to reality, not a detail dispute.** Orchestrator built an elaborate desync-via-mouse theory and held it across multiple corrections. Each pushback was treated as needing more nuance. The right move was to drop the theory and re-listen to the symptom. Saved as `feedback_listen_when_user_says_off_track.md`. Generalizes to: when a user repeats the same complaint with different words, the model probably has the wrong direction вАФ re-anchor on the literal symptom before adding theory.

2. **Instrument before fixing вАФ twice.** PR #16 made the hotkey path visible. PR #17 made the WordBuffer reset paths visible with `prev_buffer` and `reason=`. Without both, PR #18's diagnosis would have been impossible. The total cost of the instrumentation PRs was ~1 hour of Sonnet time + 2 daemon restarts; the diagnosis they unblocked saved ~half a session of guessing. **Default to instrumentation before behavior change** when the path is silent.

3. **Read-only enumeration agents are a cheap superpower.** The 272-pair table cost ~1 minute of Sonnet time to enumerate end-to-end. Knowing the *full set* of false-positives (not just the ones the user happened to encounter) reframed the problem from "this specific pair is bad" to "the entire 2-letter validator output is bad enough to gate by trust signal". The trust-model conclusion came directly from looking at the table.

4. **Advisor's role: distinguish two-hypothesis spaces.** Two calls to advisor. First (early): pointed out the bug had two distinct sides (validator + desync), needed to be fixed separately. Second (mid): "your hypothesis is sound but two explanations fit the same symptom" вЖТ clarified the two competing mechanisms (external desync vs internal reset) and what instrumentation would discriminate. Advisor is most useful when the orchestrator has converged on a single theory and is about to act on it.

5. **`gh pr create` defaults to upstream when origin is a fork.** Caught one PR mis-route (PR #18 вЖТ moiz2306/layout-switcher#2 instead of dslabakov/layout-switcher#18). Pattern of failure for any forked-repo orchestrator-pattern project on this Mac. Save explicit `--repo dslabakov/layout-switcher` boilerplate in `docs/reference/DELEGATION.md` for next session.
