## 1. Architectural map

The app is a macOS menu-bar daemon. A Python `NSApplication` hosts a status-bar icon, a settings window, and a global keyboard monitor. The event tap and detection logic run on a background thread; UI interactions run on the AppKit main thread. There is no IPC — all components share in-process state via object references passed at construction time.

```
src/
├── main.py              — NSApplication setup, permission check, thread launch
├── keyboard_monitor.py  — CGEventTap callback + word-complete detection + hotkey
├── word_buffer.py       — keystroke accumulator, boundary detection
├── layout_mapper.py     — bidirectional QWERTY↔ЙЦУКЕН character tables
├── language_detector.py — pymorphy3 (RU) + file wordlist (EN) classification
├── auto_corrector.py    — synthetic CGEvent injection (backspaces + retype)
├── correction_tracker.py— in-memory daily counter + recent-10 deque (thread-safe)
├── app_filter.py        — per-app exclusion via NSWorkspace.frontmostApplication
├── config.py            — YAML read/write with observer callbacks
├── status_bar.py        — NSStatusItem + NSMenu (NSMenuDelegate pattern)
├── settings_window.py   — NSTabView settings UI (lazy singleton)
└── onboarding_window.py — step-by-step permission wizard with NSTimer polling
```

**Event-tap loop:** `keyboard_monitor.py:65–91`. `KeyboardMonitor.start()` runs on a daemon thread (`main.py:78–79`). It creates a `CGEventTap` for `kCGSessionEventTap | kCGHeadInsertEventTap`, registers it on `CFRunLoopGetCurrent()` (the background thread's run loop), then calls `CFRunLoopRun()` to block.

**Detection pipeline:** `_tap_callback` (tap thread) → `_detection_queue` (thread-safe `queue.Queue`) → `_detection_worker` (a second daemon thread, `keyboard_monitor.py:155–178`). Classification and correction happen in the worker, not in the tap callback, to avoid blocking the event tap.

**RU/EN classification:** `language_detector.py`. Russian: `pymorphy3.MorphAnalyzer`, restricted to `DictionaryAnalyzer` results only (no guess morphology). English: plain set lookup in `data/en_wordlist.txt` (~234k words built from `/usr/share/dict/words` + tech terms at install time).

**pymorphy3 plug-in point:** `language_detector.py:13`. MorphAnalyzer is initialized at construction; `is_russian()` calls `self._morph.parse()` on each candidate. A warmup call at line 14 preloads the dictionary data to avoid latency on the first real word.

**Menu-bar UI ↔ corrector interaction:** Shared object references only. `StatusBar` and `SettingsWindow` both hold a `Config` reference. `Config` has an observer list (`config.py:28–29`). When settings change, observers fire synchronously on whatever thread called the setter — currently only `language_detector.reload_ignore_words`. The `config.enabled` flag is checked inside `app_filter.should_process()` which is polled in the tap callback; there is no signal to pause the worker thread.

## 2. Behavior summary

**Happy path: user types `ghbdtn` then presses space.**

1. **Keydown events arrive at `_tap_callback`** (`keyboard_monitor.py:97`). Each `kCGEventKeyDown` event checks `SYNTHETIC_MARKER_FIELD` (field 42, `auto_corrector.py:22–23`) to skip self-injected events (line 114). Each letter character is extracted via `CGEventKeyboardGetUnicodeString` (line 241–246) and fed to `_word_buffer.add_char(char)` (line 146).

2. **`WordBuffer.add_char`** (`word_buffer.py:16–25`) appends to `_buffer` for non-boundary chars. When space (a `BOUNDARIES` member) arrives, it returns `("ghbdtn", " ")` and clears the buffer.

3. **Back in `_tap_callback`** (line 146–151): the word meets the `len >= 2` threshold, is not in the ignore list, and passes `_could_be_word` (all chars are alpha or in `LAYOUT_LETTER_KEYS`). The tuple `("check", ("ghbdtn", " "))` is enqueued.

4. **`_detection_worker`** (`keyboard_monitor.py:155–178`) dequeues the item. It calls `_is_stale()` (line 248–250) — if the queue has more items, the word is skipped because backspaces would hit the wrong position. Otherwise `_check_and_correct("ghbdtn", " ")` runs.

5. **`_check_and_correct`** (`keyboard_monitor.py:180–210`) calls `_try_detect("ghbdtn")`. `_try_detect` calls `layout_mapper.convert_word("ghbdtn")` → `("ghbdtn", "привет")`, then `language_detector.check("ghbdtn", "привет")`. The detector confirms `ghbdtn` is not English and `привет` is Russian → returns `"correct"`. `_check_and_correct` gets `("ghbdtn", "привет")`, calls `_convert_boundary(" ", "ghbdtn")` → space stays space (space is not in the mapping tables), then calls `auto_corrector.correct("ghbdtn", "привет", " ", extra="")`.

6. **`AutoCorrector.correct`** (`auto_corrector.py:90–109`) acquires `_lock`, sets `_is_correcting = True`. Sends 7 backspaces (`len("ghbdtn") + len(" ") + len("") = 7`) via `_send_backspaces`, sleeps `BLOCK_DELAY` (8 ms), types `"привет"` then `" "` using zero-keycode `CGEventKeyboardSetUnicodeString` events. Each synthetic event is tagged with field 42 = `0x4C53` so the tap callback recognizes and passes them through. Sets `_last_correction` for undo tracking.

**Edge cases handled by commit `2e7ff6c`:**

- **Self-interception:** `keyboard_monitor.py:112–115`. The tap checks `SYNTHETIC_MARKER_FIELD` before any other processing. If it matches `SYNTHETIC_MARKER_VALUE`, the event is returned immediately without buffering, preventing the corrector from seeing its own output.

- **Fast typing (replay buffer):** `keyboard_monitor.py:117–121`. If `is_correcting` is True when a new real key arrives, the character is added to `_replay_buffer` instead of `_word_buffer`, and the event is suppressed (returns `None`). After `correct()` finishes, `_detection_worker` drains the replay buffer (line 170–178), retypes each char via `_type_string`, and feeds them back into `_word_buffer` to keep detection state in sync.

- **Punctuation mapping:** `keyboard_monitor.py:223–228`. When a word boundary is a character like `"/"` (which maps to `"."` on ЙЦУКЕН), `_convert_boundary` runs it through `layout_mapper.convert()` so the re-injected boundary is correct for the target layout.

- **Trailing layout-mapped chars:** `_check_and_correct` lines 194–210. If the full word fails detection (e.g., `"ghbdtn,"` → `"приветб"` — not a real word), it strips trailing `LAYOUT_LETTER_KEYS` chars one at a time and retries. First success wins; the trailing chars are re-injected as typed.

## 3. Code quality observations

### Real bugs

**[BUG 1] Hotkey is hardcoded; `config.hotkey_convert` is silently ignored.**
`keyboard_monitor.py:256–259`: `_is_hotkey` compares directly against `kCGEventFlagMaskControl | kCGEventFlagMaskShift` and keycode 49 (space). The value `config.hotkey_convert` is read only in `settings_window.py:127` to display it as a label — it is never parsed or wired into detection logic. Changing the config file or UI has no effect on what hotkey actually triggers. The README says the hotkey is `Ctrl+Shift+Space`; that is accurate, but it implies user-configurability that does not exist.

**[BUG 2] `config.show_notifications` is stored but never consumed.**
Set by `settings_window.py:237–239`, persisted via `config.set_show_notifications()`, but there is no code path that reads `config.show_notifications` to actually display a notification. README acknowledges "coming soon." The UI toggle is live but the feature behind it is a stub. This is not a crash, but it is silent no-op behavior that will confuse a user who enables it.

### Threading model and race conditions

**Thread structure:** Three threads are active during normal operation.
1. Main thread — AppKit run loop (`NSApplication`), menu-bar events, settings window actions.
2. Tap thread — `CFRunLoopRun()` in `KeyboardMonitor.start()`, running the `_tap_callback`. This thread also registers the `NSWorkspace` app-activation observer (`keyboard_monitor.py:66–68`).
3. Worker thread — `_detection_worker`, consuming `_detection_queue`.

**[FRAGILITY] `_appDidActivate_` observer registered from background thread.** `keyboard_monitor.py:66–68` calls `NSWorkspace.sharedWorkspace().notificationCenter().addObserver_selector_name_object_` from the tap thread (the daemon thread, not the main thread). `NSWorkspace` notifications are normally posted on and delivered to the main thread. An observer registered on a non-main thread that subsequently enters its own `CFRunLoop` may not receive these notifications. In practice this could mean app-switch events (which clear `_word_buffer` and invalidate undo) silently fail to fire, leaving stale buffer state after switching apps.

**[FRAGILITY] `_word_buffer` shared without a lock.** `_word_buffer` is written by both the tap callback (line 146) and the detection worker (line 173, replayed chars). Python's GIL prevents byte-level corruption, but logical races are possible: the worker is mid-replay (feeding chars back through `add_char`) while the tap fires a new real keystroke also calling `add_char`. These interleave in arbitrary order.

**[FRAGILITY] `_last_completed_word` updated from tap callback only (lines 148–149) but read in `_handle_hotkey` (line 235) from the worker thread.** Same GIL caveat applies.

**[FRAGILITY] Replay buffer race window.** `AutoCorrector._is_correcting` is set to `False` in the `finally` block at line 109 before the worker drains `replay_buffer` at line 170. New real keystrokes in the gap after `_is_correcting` flips to `False` but before the worker reads the queue item go directly to `_word_buffer` via the tap callback (line 146), while the worker is also writing to `_word_buffer` via the drain loop (line 173). The operations interleave.

**[MINOR] `replay_buffered` at `auto_corrector.py:81–88` is unreferenced.** It was superseded by the `drain_replay_buffer()` + `_type_string` pattern in the worker. Dead code, not harmful.

### Unhandled exception surface

`CFRunLoopRun()` must never propagate an exception; if `_tap_callback` raises, the tap is silently disabled or the process crashes. There is no `try/except` around the tap callback body. The detector and mapper calls are unlikely to raise, but `CGEventKeyboardGetUnicodeString` with a bad buffer length would. Low probability, but unguarded.

`KeyboardMonitor.start()` raises `RuntimeError` if `CGEventTapCreate` returns `None` (line 84–85). This propagates into the daemon thread's stack and is not caught anywhere; it would terminate the thread silently without updating the status-bar to error state, since `status_bar.set_error()` already ran (or didn't) based on the pre-flight check at `main.py:72`.

### Accessibility permission revocation

If permissions are revoked at runtime, the tap receives `kCGEventTapDisabledByTimeout` (`keyboard_monitor.py:98–101`) and re-enables itself. This handles the *timeout* case but not the *permission-revoked* case. CGEventTap disabled due to revoked Accessibility produces a different event type or simply stops delivering events; there is no code path that re-checks `CGPreflightListenEventAccess` / `CGPreflightPostEventAccess` and updates the menu-bar icon to orange after startup. The app would silently do nothing rather than show an error state.

### Blocking calls on the main thread

`settings_window.py:259–262`: `subprocess.run(["launchctl", "load", ...])` and `subprocess.run(["launchctl", "unload", ...])` are called synchronously from `_toggle_launch_at_login`, which fires on the main thread in response to a UI toggle. This blocks the AppKit run loop for the duration of the launchctl round-trip. Unlikely to be noticeable (launchctl is fast), but is a pattern to be aware of.

The same function is non-atomic: it writes the plist, then loads it, then calls `config.set_launch_at_login()`. A failure at the `launchctl load` step leaves the plist written but the config flag unset, leaving the "Launch at login" toggle in an inconsistent state relative to filesystem reality.

### Logging

Only `logging.INFO` level is used at runtime. `logger.debug` calls appear in `keyboard_monitor.py:167` (stale skip) but there is no way to enable debug output without editing source. The `config.log_errors` flag only gates whether a file handler is added; it does not affect the log level. There is no diagnostic mode (e.g., `--debug` flag or config key) that would log classification decisions — which words were checked, what the detector returned, why a correction was or was not applied.

### Upstream merge risk

The codebase is a clean module decomposition with clear single-responsibility boundaries. The modules least likely to conflict with upstream are `layout_mapper.py` (pure data), `word_buffer.py`, `config.py`. The highest-risk files for merge conflicts are `keyboard_monitor.py` (most logic) and `auto_corrector.py` (synthetic event injection). Both contain substantial additions (replay buffer, stale detection, synthetic marker) that diverge from whatever upstream has. Document these in `DECISIONS.md` before the first upstream sync.

## 6. Risks and sharp edges

- **Event tap can freeze keyboard input system-wide.** If `_tap_callback` hangs (e.g., `pymorphy3` init on first call blocks, or a lock deadlock in `AutoCorrector`), all keyboard input on the machine stalls until the process is killed. Always run foreground (`python3 -m src.main`) to validate changes before reloading the LaunchAgent.

- **`kCGHeadInsertEventTap` requires Accessibility permission.** The app requests both Input Monitoring (`kCGSessionEventTap` listen) and Accessibility (event posting). If Accessibility is revoked mid-session, `CGEventPost` calls in `AutoCorrector` silently fail — backspaces and retype events are not delivered. The word is partially deleted or not at all. There is no error detection around `CGEventPost` return values.

- **`launchd KeepAlive: true` restarts the process on crash.** This is correct for a daemon but means a startup crash (e.g., missing venv, bad plist placeholder) results in rapid respawn loop. Check `~/.config/layout-switcher/stderr.log` first on startup issues.

- **`config.hotkey_convert` looks editable but is not.** Changing it in `~/.config/layout-switcher/config.yaml` or the UI has no effect on the actual hotkey (`keyboard_monitor.py:256–259` hardcodes ctrl+shift+space). This is a silent no-op that can waste debugging time.

- **English wordlist is built at install time from `/usr/share/dict/words`.** On fresh macOS installs without Xcode Command Line Tools, `/usr/share/dict/words` may not exist. `build_wordlist.py` handles this silently (the `if dict_path.exists():` guard), producing a 100-entry tech-only list with no warning. The detector would fail to recognize common English words, causing false corrections on any English word not in `tech_terms`.

- **Settings window calls `launchctl` on the main thread.** `_toggle_launch_at_login` at `settings_window.py:245–263` runs synchronously. It also reads `__file__` to resolve the project directory, which is the installed `.venv` python path — this correctly resolves to the repo root via `parent.parent`. But if the `.venv` Python is moved (e.g., after reinstall to a different path), the plist generated by the UI toggle will contain wrong paths while `setup.sh`-generated one would be correct.

## 7. Recommended first delegation

**Wire up `show_notifications` (Bug #2 — S-sized, additive, zero tap risk).**

The config key, the UI toggle, and the save/load path are all in place. The only missing piece is a call to `NSUserNotificationCenter` or `UNUserNotificationCenter` inside `KeyboardMonitor._check_and_correct` after a successful correction. This touches neither the CGEventTap callback nor `auto_corrector.py`. The task is fully specified (read `config.show_notifications`, if true post a notification with `original → corrected`), the scope is one new function call in `keyboard_monitor.py`, and the result is immediately verifiable by the user without permission concerns or timing sensitivity.
